## 📖 私のストーリー：「月額固定なのに止まる」現象から解放されるまでの道のり

### Before：Max プランの「見えないコスト」に消耗する日々

正直に告白します。私は当時、こんな状況でした。

- Claude Code **Max プラン（月 1.5 万円）** をフル活用
- それでも **5 時間ごとの利用制限**にしばしば引っかかる
- 朝はサクサク動いていたモデルが **昼過ぎから明らかに性能劣化**（同じ質問への回答が雑になる）
- 「**待ち時間で消える 1-2 時間 / 日 × 平日 20 日 = 月 30 時間の損失**」という現実
- 機密性の高い案件は外部 API に投げにくく、結局自分で書く

月額は固定 1.5 万円ですが、**「お金を払ってるのに進捗が止まる」** という別種のストレスが膨らんでいました。

「**この時間損失をハードウェアでカバーできないか?**」と考え始めたのが、ローカル LLM への移行の事の発端です。

### 転機：RTX 5090 (32GB VRAM) との出会い

NVIDIA RTX 5090 を導入。VRAM 32GB は、ついに 27B〜30B クラスのモデルを **量子化なしに近い品質**で動かせる規模になりました。

- 27B モデル = LLaMA 3.3 / Qwen3.6 / Gemma 3 など、商用 GPT-3.5 を凌ぐ性能のレンジ
- 量子化 Q4_K_M なら 17GB → 32GB VRAM に余裕で乗る

そこで選んだのが **Qwen3.6-27B Q4_K_M** です。Apache 2.0 ライセンスで商用利用も自由、日本語の品質も Llama より明らかに良い。

### After：6 種類のコーディングタスクで実測した結果

結論から言うと、**Q4_K_M は「サクサク使える日常アシスタント」のレンジ** に確実に到達していました。

## 🛠 セットアップ手順（5 ステップ）

### Step 1. Ollama インストール

```powershell
winget install --id Ollama.Ollama -e
```

### Step 2. モデル取得（17GB ダウンロード）

```powershell
ollama pull qwen3.6:27b-q4_K_M
```

私の環境（光回線）では **約 4 分**で完了しました。

### Step 3. Python 環境

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install openai rich prompt_toolkit pynvml
```

### Step 4. CLI 起動

OpenAI 互換 API 経由で接続するだけ。

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:11434/v1",
    api_key="ollama",  # 任意の値で OK
)

resp = client.chat.completions.create(
    model="qwen3.6:27b-q4_K_M",
    messages=[
        {"role": "system", "content": "あなたは熟練のエンジニアです。\n/no_think"},
        {"role": "user", "content": "Pythonで リストを反転する一行コードを示してください。"},
    ],
)
print(resp.choices[0].message.content)
```

### Step 5. ⚠️ 必須：思考モードを OFF にする

Qwen3 系は **デフォルトで thinking モードが ON**。コーディング用途では遅くなるだけなので必ず OFF にしてください。

```python
# system プロンプトの末尾に /no_think を追加するだけ
"あなたは熟練のエンジニアです。\n/no_think"
```

私の環境では、これで **応答時間が 60 秒 → 20 秒（3 倍速）** になりました。

| 月のトータル | 15,000 円 + 待ち時間ストレス | **1,500 円** |

**直接的な月額差は 13,500 円**。「年 16 万円の節約」だけ見ると微妙に思えるかもしれませんが、本当の価値は別にあります:

| 隠れた価値 | 影響 |
|---|---|
| **5h 制限なし** | 集中できる夜中・休日の連続作業が可能 |
| **性能が常に一定** | 「今日の Claude は調子悪い」がない |

**ハードウェア投資の回収期間（直接費用のみ）**:
- 50 万円 ÷ 1.35 万円/月 = **約 37 ヶ月（3 年）**

ただし「待ち時間損失 月 30 時間 × 時給換算」を加味すると、**1 年以内で回収**する計算になります（時給 5,000 円換算で月 15 万円相当の損失回避）。

## 🎯 Coding 精度：75% の動くコード生成（実行テストで判定）

「キーワードが含まれているか」ではなく **「実行して動くか」** で判定した結果:

| 指標 | Q4_K_M |
|---|---|
| **実行可能テスト合格率** | **75% (8 タスク中 6)** |
| 構文エラー率 | **0%** (24 ケース全て構文OK) |
| 平均生成時間 | **22.4 秒** |
| 平均コード品質スコア | 0.92 |

具体例 (FizzBuzz):

```python
def fizzbuzz(n: int) -> str:
    if n % 15 == 0:
        return "FizzBuzz"
    elif n % 3 == 0:
        return "Fizz"
    elif n % 5 == 0:
        return "Buzz"
    else:
        return str(n)
```

→ 教科書的な書き方を、22 秒で生成。型ヒントも入っており、即本番投入可能なレベル。

**驚きの事実**: Q4 と Q8 で **生成コードがほぼ同一**（FizzBuzz・dataclass リファクタ・例外処理は完全一致）。**コーディング用途で Q8 を使う意味は皆無**。

詳細は別記事 **【Coding 精度編】**で全 8 タスク × 24 コードサンプルを公開。

## 🛠 Tool Calling 精度：80% 完璧 / 90% ツール選択正解

「Claude Code 風の **エージェント的な使い方**ができるか?」が、ローカル LLM 移行の最大の関門。

別記事 **【Tool Calling 編】** で詳細に検証しましたが、Q4_K_M の数値だけ抜粋すると:

| 指標 | Q4_K_M |
|---|---|
| 平均 score（10 タスク）| **0.85 / 1.00** |
| 完璧率（ノーミス）| **80%** |
| 正しいツール選択率 | **90%** |
| JSON 妥当率 | **100%** |
| 平均応答時間 | **3.9 秒/呼び出し** |

特筆すべきは:
- **「config.py の 10 行目から 30 行目」** という日本語から `start_line: 10, end_line: 30` を **数値型で正確に抽出**

- **JSON 妥当率 100%**（30 ケース全てパース成功）

つまり Q4_K_M は **Claude Code の代替として実用レベル**です。詳細は別記事で。

## ⚠️ ハマりポイントと回避策

### 1. PowerShell の文字化け

```powershell
$env:PYTHONIOENCODING = "utf-8"
```

を起動時に必ずセット。日本語応答が文字化けで読めなくなります。

### 2. Ollama 起動忘れ

別ターミナルで `ollama serve` を立ち上げ忘れると、エラーが出続けます。死活確認:

```powershell
Invoke-RestMethod http://localhost:11434/api/tags
```

### 3. Q4 で「これ無理かな」と思ったら

実際に試して品質が足りないなら **Q5_K_M (20GB) に上げる**選択肢があります。Q5_K_M は Ollama 公式にはありませんが、HuggingFace の bartowski さんが量子化版を公開しているので、Modelfile を自作して登録できます。

→ 詳細は次回の **【Q5_K_M 編】** で解説します。
